## Section 1 — Proposed Gate 6D Capability

### Candidate Capabilities (Ranked)

#### 🥇 **RECOMMENDED: Export Delivery History & Tracking (Gate 6D-A)**

**Capability Description:**
Track, retrieve, and audit all historical report exports generated by MGA users. Provides visibility into who exported what data, when, and for what purpose.

**Key Features:**
- Export history dashboard showing all past exports by user, date, report type, format
- Download artifact retrieval (re-download previously generated reports)
- Export timeline with actor, timestamp, scope, and outcome
- Filter/search by date range, user, report type, case
- Audit trail integration (view who accessed which exports)

**Why Recommended as Gate 6D:**
- ✅ Direct dependency on Gate 6C (exports must exist to have history)
- ✅ Low implementation complexity (UI + queries over existing audit logs)
- ✅ No new data model required (uses ActivityLog entity)
- ✅ No new permissions needed (leverages existing `mga.reports.audit` key)
- ✅ Minimal scope expansion (read-only history/audit view)
- ✅ High operational value (compliance + visibility)
- ✅ Zero risk to Gates 6A/6B (no interaction with user management or transmit)
- ✅ Clear deferral path if Gate 6C remains inactive

**Risks (Managed):**
- ⚠️ MEDIUM: Audit log query performance (large export volumes)
  - *Mitigation:* Pagination + indexing on ActivityLog
- ⚠️ LOW: Data leakage from cached artifacts
  - *Mitigation:* Signed URL expiration; scope validation on re-download

---

#### 🥈 **ALTERNATIVE: Saved Report Templates (Gate 6D-B)**

**Capability Description:**
Allow MGA users to save custom report configurations (filter presets, field selections) as reusable templates.

**Key Features:**
- Create/save/edit/delete report templates
- Template management dashboard
- Quick-generate from template
- Share templates within MGA scope
- Template versioning

**Why Not Recommended (for Gate 6D):**
- ❌ Requires new entity (ReportTemplate)
- ❌ Requires new permissions (template create/edit/delete)
- ❌ Scope management complexity (who can share templates?)
- ❌ Better deferred to Gate 6E (after 6D foundation established)

---

#### 🥉 **ALTERNATIVE: Report Scheduling (Gate 6D-C)**

**Capability Description:**
Schedule recurring report generation (daily, weekly, monthly) with automatic delivery.

**Key Features:**
- Schedule configurations (cron-like interface)
- Scheduled export history
- Automatic delivery (email, storage)
- Pause/resume/cancel schedules

**Why Not Recommended (for Gate 6D):**
- ❌ Requires job scheduler integration (complex)
- ❌ Requires notification/delivery infrastructure
- ❌ Placeholder storage model insufficient (needs cloud storage)
- ❌ Better Phase 2+ enhancement (after Gate 6C fully stabilized)

## Section 4 — Risk Review

### Risk Category: Scope Leakage

**Risk Statement:**
Gate 6D history queries could expose export records across MGA boundaries.

**Mitigation Strategy:**
- ✅ ScopeGate validation on all ActivityLog queries
- ✅ MGA ID filter on all result sets
- ✅ Scope-aware pagination (filter before paginating)
- ✅ Server-side scope validation (not client-side)

**Risk Level:** 🟢 LOW (proven pattern from Gate 6C)

---

### Risk Category: Permission Bypass

**Risk Statement:**
User could view export history of other MGA users or restricted reports.

**Mitigation Strategy:**
- ✅ Role-based permission keys enforced
- ✅ mga_user role explicitly denied access
- ✅ Manager role denied audit details (summary only)
- ✅ Centralized permission resolver (no hardcoded checks)

**Risk Level:** 🟢 LOW (same pattern as Gate 6C)

---

### Risk Category: Data Leakage Through History

**Risk Statement:**
Audit log history could expose sensitive data (PII, field values, etc.).

**Mitigation Strategy:**
- ✅ Audit entries log action/actor/outcome only (Gate 6C pattern)
- ✅ Sensitive fields excluded from audit logging
- ✅ PII masked in audit details
- ✅ History view sanitizes audit display

**Risk Level:** 🟢 LOW (inherits Gate 6C security)

---

### Risk Category: UI Exposure Risk

**Risk Statement:**
History UI could be accidentally exposed in production while Gate 6C is inactive.

**Mitigation Strategy:**
- ✅ History UI depends on `MGA_REPORT_EXPORTS_ENABLED` flag
- ✅ If flag is false, history tab hidden automatically
- ✅ Backend rejects history queries if flag is false
- ✅ No history data available until Gate 6C activated

**Risk Level:** 🟢 LOW (fail-closed via feature flag)

---

### Risk Category: Regression to Gates 6A, 6B, 6C

**Risk Statement:**
Gate 6D implementation could break user management, quote transmission, or export functionality.

**Mitigation Strategy:**
- ✅ Read-only access to Gate 6C data (no mutations)
- ✅ Zero interaction with Gate 6A (user management)
- ✅ Zero interaction with Gate 6B (transmit)
- ✅ New files only; no modification to existing gates
- ✅ Comprehensive regression test suite (40–50 tests)

**Risk Level:** 🟢 VERY LOW (isolated scope)
