### Gate 6D Dependency

> **Gate 6D must remain inactive until Gate 6C is activated and its post-activation smoke validation passes.**  
> Gate 6D tracks export history records that are generated by Gate 6C operations. Without Gate 6C live and producing export audit events, Gate 6D has no data to display and its smoke test cannot be meaningfully executed.  
> `MGA_EXPORT_HISTORY_ENABLED` must remain `false` regardless of this packet's outcome.

## Section 2 — What Activation Would Change

Setting `MGA_REPORT_EXPORTS_ENABLED = true` would produce the following and **only** the following changes:

| Change | Detail |
|--------|--------|
| Export modal becomes visible | `MGAReportExportModal` is rendered for users whose role passes `canExport` (mga_admin, platform_super_admin only per permission matrix) |
| Export backend becomes reachable | `mgaReportExport` backend function no longer returns `503 Feature not enabled`; it proceeds to scope and permission evaluation |
| Audit events begin generating | Successful and failed export attempts are written to the audit log via `reportExportAudit.js` |
| Field policy enforcement becomes active | `reportExportFieldPolicy.js` enforces the safe field allowlist and excludes restricted fields on every export response |

### What Activation Does NOT Change

| Item | Status After Activation |
|------|------------------------|
| Permission matrix | **Unchanged** — export is still gated by `permissionResolver.check(role, 'mga', 'export')` |
| scopeGate | **Not bypassed** — every export request passes through `scopeGate.check()` before execution |
| Cross-MGA data isolation | **Unchanged** — export results are filtered by `effective_mga_id` from actor membership |
| Gate 6D authorization | **Not granted** — `MGA_EXPORT_HISTORY_ENABLED` remains `false`; Gate 6D requires separate operator approval |
| Gate 6C closure status | **Not changed** — Gate 6C is not marked CLOSED until post-activation validation passes and operator provides final sign-off |
| Gate 6A behavior | **Unchanged** |
| Gate 6B behavior | **Unchanged** |

## Section 4 — Gate 6C Activation Procedure

The following sequence must be executed in order. No step may be skipped. No step may be executed before the operator completes Section 3 with an APPROVE decision.

| Step | Action | Owner | Confirmation Required |
|------|--------|-------|-----------------------|
| 1 | Confirm rollback owner is assigned and on record (Section 3) | Operator | Written confirmation in this packet |
| 2 | Confirm smoke test executor is assigned and on record (Section 3) | Operator | Written confirmation in this packet |
| 3 | Confirm activation window — ensure rollback owner is available during window | Operator | Written confirmation in this packet |
| 4 | Set `MGA_REPORT_EXPORTS_ENABLED = true` in `components/mga/MGACaseWorkflowPanel.jsx` | Platform Engineering | Only after Steps 1–3 complete |
| 5 | Run Gate 6C post-activation smoke test per checklist in Section 5 | Smoke Test Executor | Document result for each checklist item |
| 6 | Verify export UI is visible only for authorized roles (`mga_admin`, `platform_super_admin`) | Smoke Test Executor | Confirm unauthorized roles do not see export button |
| 7 | Verify backend export actions remain permission-gated and scope-gated | Smoke Test Executor | Confirm scopeGate denial for out-of-scope requests |
| 8 | Verify cross-MGA and cross-tenant export attempts fail closed | Smoke Test Executor | Confirm 403/denied response for cross-MGA actor |
| 9 | Verify audit events are written for export requests, successes, and failures | Smoke Test Executor | Confirm records in audit log entity |
| 10 | Verify rollback by setting `MGA_REPORT_EXPORTS_ENABLED = false` | Rollback Owner | Confirm export UI disappears; backend returns 503 |
| 11 | If all smoke validation items PASS: restore flag to `true` and update registry status to `ACTIVATED_VALIDATION_PASSING` | Platform Engineering | Document in registry and this packet |
| 12 | Gate 6C may only be marked `CLOSED` after final written operator sign-off on smoke test results | Operator | Final sign-off document required |

> **If any smoke test item FAILS:** execute rollback immediately (Section 6), document failure, and do not proceed to Step 11. Gate 6C remains `INACTIVE` until failure is diagnosed and resolved.

## Section 7 — Gate 6D Dependency Warning

> **Gate 6D must not be activated simultaneously with Gate 6C, or before Gate 6C activation smoke validation passes.**

| Constraint | Detail |
|-----------|--------|
| Gate 6D depends on Gate 6C export behavior | Gate 6D reads audit event records produced by Gate 6C export operations. Without Gate 6C live, there are no history records to display. |
| Gate 6D smoke test requires Gate 6C data | Gate 6D smoke test item ST-04 (`history records present from Gate 6C exports`) cannot pass unless Gate 6C has been live and generating export audit events. |
| `MGA_EXPORT_HISTORY_ENABLED` must remain `false` | Regardless of the outcome of this approval packet, Gate 6D's flag must not be changed. Gate 6D requires its own separate operator approval packet, issued only after Gate 6C smoke validation is declared PASS. |
| Activation sequence is strictly sequential | Gate 6C activation → Gate 6C smoke validation PASS → Gate 6D operator approval → Gate 6D activation. These steps may not be collapsed or reordered. |
