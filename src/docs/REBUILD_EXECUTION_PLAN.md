# Rebuild Execution Plan

## Phase 1 — Foundation Extraction & Contracts
1. Freeze current app as legacy reference.
2. Preserve entity schemas and audit current route map.
3. Define canonical module boundaries.
4. Create workflow contract docs and state models.
5. Define normalized audit/version entities.

## Phase 2 — Core Domain Services
1. Build shared workflow/state services.
2. Build dependency registry service.
3. Build pricing execution service.
4. Build quote traceability service.
5. Build exception generation service.

## Phase 3 — Core Operational Modules
1. Rebuild Employers module on clean services.
2. Rebuild Cases module as workflow shell.
3. Rebuild Census module on canonical snapshot/readiness services.
4. Rebuild Plans & Rates module with normalized pricing governance.

## Phase 4 — Decision & Execution Modules
1. Rebuild Quotes module as deterministic decision engine.
2. Rebuild Proposal module with quote lineage.
3. Rebuild Enrollment module with approved quote mapping.
4. Rebuild Renewals with rollover and continuity services.

## Phase 5 — Operations, Admin, Integrations
1. Rebuild Exceptions module with policy-driven automation.
2. Rebuild Dashboard from shared projections.
3. Rebuild Settings/Admin around true policies/config.
4. Rebuild Integration Infrastructure around executable adapters.

## Phase 6 — Portals & Reporting
1. Rebuild employer and employee portals from normalized execution data.
2. Build reporting/export services.
3. Add payroll, billing, carrier, and EDI execution layers.

## Phase 7 — Validation & Cutover
1. Validate route integrity.
2. Validate workflow integrity.
3. Validate financial determinism.
4. Validate audit/version lineage.
5. Validate role boundaries.
6. Migrate users to rebuilt surfaces.

## Priority Build Order
1. Shared domain services
2. Cases
3. Census
4. Rates
5. Quotes
6. Proposals
7. Enrollment
8. Renewals
9. Exceptions
10. Dashboard
11. Integrations
12. Portals
13. Reporting

## What To Preserve
- business intent
- canonical entities where still valid
- current workflow concepts
- useful UI patterns
- route vocabulary users already know

## What To Replace
- page-local domain logic
- inline JSON-only audit/version strategies
- partial downstream mappings
- duplicated readiness logic
- weak integration placeholders
- control planes without real orchestration behind them

## Success Criteria
- every workflow has explicit states and transitions
- all financial outputs are deterministic
- all critical changes are audited
- quotes, enrollment, and renewals share the same financial core
- exceptions are automatically generated by policy
- integrations are adapter-driven and failure-aware
- UI pages are thin orchestration shells, not business logic owners
