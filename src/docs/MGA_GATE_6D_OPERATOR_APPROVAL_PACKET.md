> **Gate 6C must be activated and its post-activation smoke validation must pass before Gate 6D activation may be considered.**

## Section 2 — What Activation Would Change

Setting `MGA_EXPORT_HISTORY_ENABLED = true` would produce the following and **only** the following changes:

| Change | Detail |
|--------|--------|
| Export History tab becomes visible | The `Export History` tab renders in the MGA Case Workflow Panel for users whose role passes `hasHistoryPermission(role, HISTORY_PERMISSIONS.VIEW)` |
| `MGAExportHistoryPanel` becomes mounted | The history panel component is rendered and begins fetching export history records via `mgaExportHistoryContract` |
| History backend becomes reachable | `mgaExportHistoryContract` no longer returns `503 Feature not enabled`; it proceeds to scope and permission evaluation |
| Audit events for history access begin generating | History view requests and denials are written to the audit log |
| Export history records become queryable | Authorized actors can retrieve the history of past export operations scoped to their MGA |

### What Activation Does NOT Change

| Item | Status After Activation |
|------|------------------------|
| Permission matrix | **Unchanged** — history access is still gated by `hasHistoryPermission()` |
| scopeGate | **Not bypassed** — every history request passes through `scopeGate.check()` before execution |
| Cross-MGA data isolation | **Unchanged** — history results are filtered by `effective_mga_id` from actor membership |
| Gate 6C authorization | **Not granted or revoked** — Gate 6C activation is independent; if Gate 6C is inactive, Gate 6D will show an empty history with no new export records being generated |
| New report export creation | **Not enabled** — Gate 6D reads existing export audit records; it does not create new exports |
| Signed URLs | **Never returned** — `reportExportHistoryPayloadPolicy.js` excludes signed URLs from all history responses |
| Private file URIs | **Never returned** — private storage URIs are excluded from all history responses |
| Exported content | **Never returned** — raw export payloads are not surfaced in history responses |
| Retry / cancel controls | **Not exposed** — retry and cancel actions are not authorized and remain hidden |
| Gate 6D closure status | **Not changed** — Gate 6D is not marked CLOSED until post-activation validation passes and operator provides final sign-off |
| Gate 6A behavior | **Unchanged** |
| Gate 6B behavior | **Unchanged** |

## Section 8 — Gate 6C Dependency Warning

> **Gate 6D must not be activated before Gate 6C, and must not be activated unless Gate 6C activation smoke validation has been declared PASS.**

| Constraint | Detail |
|-----------|--------|
| Gate 6D depends on Gate 6C export operations | Gate 6D reads audit event records produced by Gate 6C export operations. Without Gate 6C live and generating export audit events, Gate 6D has no history records to display and its smoke test cannot be meaningfully validated. |
| Gate 6D must not be activated simultaneously with Gate 6C | The two gates must be activated sequentially with a full smoke validation pass for Gate 6C before Gate 6D activation begins. |
| Gate 6C rollback triggers Gate 6D mandatory response | If Gate 6C is rolled back after Gate 6D is active, Gate 6D must either remain inactive (if not yet activated) or be rolled back immediately. A rolled-back Gate 6C with an active Gate 6D creates a broken state — history panel active with no new export events being generated. |
| Activation sequence is strictly sequential | Gate 6C activation → Gate 6C smoke validation PASS → Gate 6D operator decision → Gate 6D activation → Gate 6D smoke validation. These steps may not be collapsed or reordered. |
| Gate 6D closure requires Gate 6C to remain active | Gate 6D may not be marked CLOSED unless Gate 6C is simultaneously active and validated. |
